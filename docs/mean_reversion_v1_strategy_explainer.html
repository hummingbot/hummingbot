mean_reversion_v1: trade decision logic, parameters, and executor outputs

This page explains the actual runtime behavior of controllers/directional_trading/mean_reversion_v1.py and the base controller path it inherits from. It covers every controller input, every strategy output that feeds execution, and the exact conditions that trigger entry, block an entry, and stop an active executor.

What this strategy is doing

It computes a rolling fair value and z-score from candles, filters those raw deviations through RSI, volatility, volume, and optional trend distance checks, then emits a directional signal:

signal = 1 -> propose long entry
signal = -1 -> propose short entry
signal = 0 -> no entry proposal
Important: a non-zero signal does not automatically create a trade. The inherited base controller still applies cooldown and max-active-executor gates before it emits a CreateExecutorAction.

1. Runtime flow

The trade path starts outside this file. The strategy-level tick loop waits until connectors are ready, then the controller loop waits until market data is ready and executors have been synchronized. Only then does the mean reversion controller compute indicators and propose actions.

Source: strategy_v2_base.py tick/on_tick
Step 1
StrategyV2Base.tick() runs every tick. If any connector is not ready, it logs "<connector> is not ready. Please wait..." and returns.
Step 2
Once connectors are ready, the strategy continues its normal tick path and the controller's async loop is allowed to do work.
Step 3
ControllerBase.control_task() checks: market_data_provider.ready, then executors_update_event.is_set(). If either is false, no decision is made on that cycle.
Step 4
MeanReversionV1Controller.update_processed_data() loads candles, computes features, and stores the latest signal in self.processed_data.
Step 5
DirectionalTradingControllerBase.determine_executor_actions() combines create_actions_proposal() and stop_actions_proposal().
Step 6
If create or stop actions are returned, the controller sends them to the strategy via actions_queue. If not, the controller stays idle and may log a periodic running but idle message with the current no-action reason.

3. Derived features

update_processed_data() is the core of the strategy. It pulls candles, optionally removes the latest in-progress candle, and computes the features that drive entry and the mean-reversion exit signal.

Source: mean_reversion_v1.py update_processed_data()
Fair value EMA(close, lookback_period) if use_ema else SMA(close, lookback_period)
Standard deviation std = rolling_std(close, lookback_period, ddof=0)
Z-score z_score = (close - fair_value) / std
Normalized volatility std_pct = std / fair_value
Trend distance trend_ema = EMA(close, trend_ema_period) trend_deviation = abs(close - trend_ema) / trend_ema
Volume ratio volume_avg = rolling_mean(volume, volume_lookback) volume_ratio = volume / volume_avg
RSI delta = close.diff() gain = EMA(max(delta, 0), rsi_length) loss = EMA(max(-delta, 0), rsi_length) rsi = 100 - (100 / (1 + gain / loss))
Close to mean close_to_mean = abs(z_score) <= exit_z_score

Why the strategy keeps a full feature DataFrame

The latest row drives the current signal, but the full DataFrame is kept in processed_data["features"] so the controller can format status, backtesting can export the whole trace, and diagnostics can inspect the full indicator history.

How many candles are requested

max_records is set to the maximum of lookback, trend EMA period, volume lookback, and RSI length, plus 25 extra rows. That gives the indicators warm-up room before the latest row is consumed.

4. Entry decisions

The strategy produces a raw directional signal in update_processed_data(). The base directional controller then turns that signal into a create action if the gating checks pass.

Long entry: signal = 1

All of these must be true on the latest feature row:

  • z_score <= -entry_z_score
  • rsi <= rsi_long_threshold
  • min_std_pct <= std_pct <= max_std_pct
  • volume_ratio >= min_volume_ratio
  • If trend filter is enabled: trend_deviation <= max_trend_deviation

Short entry: signal = -1

All of these must be true on the latest feature row:

  • z_score >= entry_z_score
  • rsi >= rsi_short_threshold
  • min_std_pct <= std_pct <= max_std_pct
  • volume_ratio >= min_volume_ratio
  • If trend filter is enabled: trend_deviation <= max_trend_deviation

No entry: signal = 0

This is the default. It means at least one required long or short condition failed, or the indicators are still warming up and produced NaN-driven false conditions.

Important distinction: the mean reversion file only computes the directional intent. The actual trade proposal is emitted by DirectionalTradingControllerBase.create_actions_proposal().

5. Entry blockers after a valid signal

Once the latest signal is non-zero, the inherited base controller still decides whether an executor may be created. This is why a backtest trace can show signal = 1 or signal = -1 without a CreateExecutorAction.

waiting_for_signal

Set when signal == 0. No create action is attempted.

max_executors_reached

Set when the number of active executors on the signal side is already equal to max_executors_per_side.

cooldown_active

Set when the most recent active executor on the same side is too recent: current_time - max_timestamp <= cooldown_time.

Market data readiness gates

The controller makes no decision at all until market_data_provider.ready is true and executors_update_event is set.

7. Exit decisions

There are two distinct exit layers: the controller-level mean-reversion stop path, and the executor-level triple barrier path.

Controller-level exit: back to mean

MeanReversionV1Controller.stop_actions_proposal() returns no stop actions unless both of these are true:

  • close_on_mean_reversion == True
  • processed_data["close_to_mean"] == True

close_to_mean is computed as abs(z_score) <= exit_z_score. When it fires, the controller emits one StopExecutorAction for every active executor.

Executor-level exit: triple barrier

Every created PositionExecutorConfig includes a TripleBarrierConfig with these fields:

  • stop_loss
  • take_profit
  • time_limit
  • trailing_stop
  • open / take-profit / stop-loss / time-limit order types

Those are not decided inside mean_reversion_v1.py at exit time. They are decided at entry time and then enforced by the executor after the position exists.

8. Strategy outputs

The strategy exposes several outputs that matter for debugging, status, and backtesting.

processed_data

Key Type Produced by Meaning
signal int update_processed_data() Latest directional intent: 1, 0, or -1.
close_to_mean bool update_processed_data() Whether the latest absolute z-score is back inside the exit band.
features DataFrame update_processed_data() Full indicator frame with candle data and computed columns.

features columns that matter

The feature DataFrame carries the original candle columns plus the strategy-derived columns: fair_value, std, z_score, std_pct, trend_ema, trend_deviation, volume_avg, volume_ratio, rsi, and signal.

get_custom_info()

This method returns a compact debug snapshot from the latest row:

{ "signal": int, "z_score": float | None, "fair_value": float | None, "rsi": float | None, "volume_ratio": float | None, "std_pct": float | None }
Bottom line: the entry decision is signal computation + base gating. The exit decision is mean reversion stop actions + executor triple barriers. That is the complete behavior you need to explain trade entries and exits in live runs or backtests.
